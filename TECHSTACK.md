# 🛠️ Выбранный технологический стек

## Обзор

Проект построен на современном JavaScript/TypeScript стеке для создания веб-приложения с визуальным редактором drag-and-drop.

---

## 🎯 Основные технологии

### 1. React 18

**Что это:**
React - популярная JavaScript библиотека для создания пользовательских интерфейсов, разработанная Facebook.

**Почему выбрана:**
- **Компонентная архитектура**: Идеально подходит для модульной структуры редактора (блоки, панели, модальные окна)
- **Reactive rendering**: Автоматическое обновление UI при изменении состояния
- **Большое сообщество**: Огромное количество готовых решений и библиотек
- **Virtual DOM**: Эффективное обновление интерфейса без лишних перерисовок
- **React Flow совместимость**: React Flow работает только с React

**Аналоги:**
- Vue.js
- Angular
- Svelte
- Solid.js

**Почему React лучше аналогов для этого проекта:**
- **Vue.js** - меньше экосистемы для специализированных библиотек (React Flow есть только для React)
- **Angular** - слишком тяжеловесный для этого проекта, избыточная сложность
- **Svelte** - меньше библиотек и сообщества, React Flow недоступен
- **Solid.js** - новее, меньше решений и документации

**Преимущества:**
✅ Компонентная структура идеальна для UI редактора  
✅ React Flow только для React  
✅ Огромная экосистема  
✅ Множество примеров и документации  

---

### 2. TypeScript

**Что это:**
TypeScript - типизированная надстройка над JavaScript с compile-time проверкой типов.

**Почему выбрана:**
- **Типобезопасность**: Уменьшает ошибки, особенно при работе с сложными структурами данных (блоки, связи, проекты)
- **Автодополнение**: Улучшает developer experience
- **Рефакторинг**: Безопасное изменение кода с проверкой зависимостей
- **Документация кода**: Типы служат документацией

**Аналоги:**
- JavaScript
- Flow (Facebook)
- ReScript
- PureScript

**Почему TypeScript лучше:**
- **JavaScript** - нет защиты от ошибок типов, сложнее отслеживать связи между компонентами
- **Flow** - менее популярен, хуже поддержка в инструментах
- **ReScript/PureScript** - слишком строгий синтаксис, высокий порог входа

**Преимущества:**
✅ Compile-time проверка ошибок  
✅ Отличная интеграция с IDE (автодополнение)  
✅ Понятная документация кода через типы  
✅ Поддержка в React Tooling  

---

### 3. React Flow

**Что это:**
React Flow - специализированная библиотека для создания интерактивных графов и диаграмм на основе React.

**Почему выбрана:**
- **Специализация**: Разработана специально для создания визуальных редакторов
- **Drag & Drop**: Встроенная поддержка перетаскивания узлов
- **Связи (edges)**: Автоматическая отрисовка связей между узлами
- **Performance**: Оптимизированная отрисовка для большого количества узлов
- **Мини-карта и контролы**: Готовые компоненты для навигации

**Аналоги:**
- D3.js (с React-D3-Tree-Graph)
- Cytoscape.js
- GoJS
- X6 (AntV)
- mxGraph (документация скудная)

**Почему React Flow лучше аналогов:**
- **D3.js** - требует много кода, сложная интеграция с React
- **Cytoscape.js** - ориентирован на сетевые графы, избыточен
- **GoJS** - коммерческий, платный лицензионный ключ
- **X6 (AntV)** - меньше документации, китайская компания (меньше англоязычных ресурсов)
- **mxGraph** - низкая документация

**Преимущества:**
✅ Создан специально для React  
✅ Готовые компоненты (Controls, MiniMap, Background)  
✅ Хорошая документация и примеры  
✅ Активная разработка и поддержка  
✅ Бесплатный open-source  

---

### 4. Zustand

**Что это:**
Zustand - минималистичная и легковесная библиотека для управления состоянием React-приложений.

**Почему выбрана:**
- **Простота**: Минимум boilerplate кода по сравнению с Redux
- **Производительность**: Оптимизированные перерисовки, перерисовываются только изменившиеся компоненты
- **Небольшой размер**: ~1 KB (минимизированный)
- **TypeScript friendly**: Отличная поддержка типов из коробки
- **Гибкость**: Легко интегрируется, нет обязательных middlewares

**Аналоги:**
- Redux (+ Redux Toolkit)
- MobX
- Jotai
- Recoil
- Context API

**Почему Zustand лучше аналогов:**
- **Redux** - слишком много boilerplate кода, избыточен для проекта этого размера
- **MobX** - более сложный, может быть избыточным
- **Jotai** - атомарный подход избыточен для простого store
- **Recoil** - избыточный для этого проекта, атомарный подход усложняет
- **Context API** - плохая производительность при частых обновлениях, нет оптимизации

**Преимущества:**
✅ Минимальный boilerplate  
✅ Хорошая производительность  
✅ Простой API  
✅ Отличная TypeScript поддержка  
✅ Легко тестировать  

---

### 5. Vite

**Что это:**
Vite - современный сборщик и dev-сервер для веб-приложений, разработанный создателем Vue.js.

**Почему выбран:**
- **Скорость**: Мгновенный HMR (Hot Module Replacement)
- **Быстрый старт**: Dev-сервер запускается за миллисекунды
- **Производительность**: Использует нативные ES-модули в dev режиме
- **Оптимизация**: Автоматическая tree-shaking для production
- **Plugins**: Легкая интеграция с React, TypeScript

**Аналоги:**
- Webpack
- Create React App (CRA)
- Parcel
- Snowpack

**Почему Vite лучше аналогов:**
- **Webpack** - медленный HMR, долгая сборка
- **Create React App** - устаревший, медленный, сложная конфигурация
- **Parcel** - медленнее Vite, менее гибкий
- **Snowpack** - менее популярен, меньше плагинов

**Преимущества:**
✅ Мгновенный HMR  
✅ Быстрая сборка production  
✅ Простая конфигурация  
✅ Современный (ESM)  
✅ Отличная TypeScript поддержка

## 📦 Дополнительные зависимости

### CSS (встроенные стили)
- Использованы CSS модули для компонентов
- Нет внешних CSS фреймворков (для минималистичного решения)

**Почему нет UI-библиотек:**
- **Bootstrap/Material-UI** - избыточны для проекта
- **TailwindCSS** - мог бы использоваться, но inline стили проще для индивидуального дизайна
- Полный контроль над стилями

### Local Storage
- Встроенный Web API для локального хранения
- Нет необходимости в дополнительных библиотеках

## 🏗️ Архитектурные решения

### Компонентная структура
```
src/
├── components/     # React компоненты
├── store/         # Zustand store
├── types/         # TypeScript типы
└── utils/         # Вспомогательные функции
```

**Почему такая структура:**
- Разделение ответственности
- Легко найти нужный код
- Масштабируемость

### State Management
- **Zustand** для глобального состояния (проекты, история, UI)
- **Local State** (useState) для компонентов с временным состоянием

**Архитектура Store:**
```typescript
interface EditorState {
  // Данные
  currentProject
  history
  
  // UI State
  selectedNodeId
  isSettingsOpen
  
  // Actions
  createProject()
  addBlock()
  // ...
}
```
